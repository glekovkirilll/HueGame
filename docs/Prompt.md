## Проект: многопользовательская браузерная игра по мотивам авторской настольной игры «Я так вижу»

## 1. Цель проекта

Нужно разработать полноценную многопользовательскую браузерную игру с одной центральной серверной инфраструктурой, в которой:

* существует несколько независимых комнат;
* у каждой комнаты есть один **хост**;
* игроки подключаются к комнате по короткому коду;
* хост обычно работает с ПК и выводит общий экран игры, например на телевизор;
* игроки обычно подключаются с телефона через браузер;
* игра идет в реальном времени;
* весь backend, frontend и постоянное хранение данных работают через **один центральный сервер**.

Система должна быть рассчитана не на локальную сессию “на коленке”, а на полноценную эксплуатацию: корректное состояние комнат, переподключение игроков, автоматическое ведение раундов, локализация, красивый и функциональный интерфейс.

## 2. Рекомендуемый стек

Нужно использовать стек, который даст:

* быстрый backend для real-time логики;
* современный красивый frontend;
* хорошую скорость разработки;
* удобную типизацию end-to-end;
* простое развертывание на одном сервере.

### 2.1 Рекомендуемый backend

Использовать **NestJS + WebSocket Gateway + PostgreSQL + Prisma**.

Обоснование:

* NestJS имеет встроенную и официально документированную модель WebSocket gateway, guards, adapters и удобен для сложной серверной бизнес-логики. ([NestJS Документация][1])
* Prisma дает type-safe доступ к БД, миграции и удобную модель разработки для TypeScript-проекта. ([Prisma][2])
* PostgreSQL — надежная реляционная БД, актуальная текущая ветка документации — PostgreSQL 18. ([PostgreSQL][3])

### 2.2 Рекомендуемый frontend

Использовать **Next.js (App Router) + React + TypeScript + Tailwind CSS + Framer Motion**.

Обоснование:

* Next.js остается актуальным фреймворком для веб-приложений на React, App Router официально поддерживается и является современным основным подходом. ([Next.js][4])
* Для красивого интерфейса, эффектов и адаптивности использовать Tailwind + Framer Motion.
* Для UI-компонентов допустимо использовать shadcn/ui или собственную дизайн-систему.

### 2.3 Transport

* Для realtime использовать **WebSocket**
* HTTP использовать для:

  * первичного входа;
  * получения статических данных;
  * healthcheck;
  * загрузки конфигурации;
  * административных действий.

### 2.4 Деплой

* Один Linux-сервер
* Docker Compose
* Сервисы:

  * `frontend`
  * `backend`
  * `postgres`
  * опционально `nginx`

## 3. Общая игровая концепция

## 3.1 Поле

Игровое поле имеет размер **30 × 15**:

* ось X: `1..30`
* ось Y: `А..О` (15 строк)

Каждая клетка:

* квадратная;
* имеет уникальную координату, например `Л8`;
* соответствует одному оттенку цвета;
* является частью гармоничной палитры.

### 3.2 Палитра

Палитра должна быть:

* визуально красивой;
* логически согласованной;
* соседние клетки должны быть похожи по оттенкам;
* желательно: центр ближе к светлым/нейтральным оттенкам, края — насыщеннее;
* палитра может **немного отличаться от игры к игре**, но должна сохранять гармоничность и читаемую структуру.

Требование:

* система должна поддерживать **генерацию палитры для новой комнаты/новой партии** на основе детерминированного seed;
* хост и игроки должны видеть согласованные координаты палитры в рамках партии;
* палитра должна храниться как часть состояния комнаты/игры, чтобы reconnect не ломал отображение.

## 4. Роли пользователей

## 4.1 Хост

Хост — это отдельный пользователь комнаты со специальным интерфейсом.

Особенности:

* подключается как хост;
* видит все игровое поле;
* не управляет вручную раундом, игра идет автоматически;
* может:

  * создать комнату;
  * настроить игру;
  * запустить игру;
  * наблюдать за ходом;
  * переподключиться как хост после обрыва.

Важно:

* хост сам по себе **не обязан быть обычным игроком**;
* но он может отдельно зайти с телефона как обычный игрок под отдельной игровой сессией.

## 4.2 Игрок

Игрок подключается по коду комнаты и уникальному имени.

Ограничения:

* от **2 до 15** игроков в комнате;
* имя в комнате должно быть уникальным;
* игрок может переподключиться под тем же именем и вернуть свою игровую сессию.

Состояния игрока:

* активный игрок;
* неактивный игрок;
* выбывший игрок;
* временно disconnected.

## 5.2 Активный игрок

Активный игрок:

* получает карточку:

  * цвет;
  * координату клетки;
  * категорию;
* устно объясняет цвет;
* не вводит подсказку в систему;
* не получает никаких фишек по итогу раунда;
* не заинтересован в результате остальных игроков.

## 5.3 Неактивные игроки

Неактивные игроки:

* видят только категорию;
* слышат устную подсказку;
* не видят поле целиком;
* используют специальный интерфейс выбора координат/клеток для ставок.

## 5.4 Ставки

Каждый неактивный игрок обязан поставить **минимум 1 фишку**, если у него есть фишки.

Правила ставок:

* одна клетка = максимум 1 фишка от конкретного игрока;
* один игрок может поставить фишки на несколько разных клеток;
* несколько разных игроков могут поставить фишки в одну и ту же клетку;
* игрок может поставить от `1` до `bank` фишек;
* часть фишек можно оставить в банке;
* если у игрока 0 фишек, он считается выбывшим и не делает ходов.

## 5.5 Подсчет

После завершения голосования:

* вокруг загаданной клетки определяется зона 5 × 5;
* каждая поставленная фишка рассматривается отдельно.

Правила множителей:

* точное попадание в центр → `x3`
* попадание в квадрат 3 × 3 вокруг центра, кроме центра → `x2`
* попадание в край зоны 5 × 5 → `x1`
* вне зоны 5 × 5 → фишка сгорает

Пример:

* игрок поставил 5 фишек:

  * 2 на край
  * 2 в радиус 1
  * 1 в центр
* результат:

  * `2 -> 2`
  * `2 -> 4`
  * `1 -> 3`
* итог возврата: `9`

## 5.6 Очередь активного игрока

* активный игрок идет по кругу;
* порядок определяется очередью входа в игру;
* новый игрок, вошедший в уже начавшуюся игру, получает стартовый банк и добавляется в конец очереди;
* если игрок disconnected, он пропускается только в том случае, если невозможно использовать его роль в текущем цикле; в остальном очередь должна сохраняться;
* если активный игрок временно вылетел, текущий раунд не ломается и продолжается, так как подсказка дана устно.

## 8. Логика таймингов

## 8.1 Начало голосования

Как только активному игроку показана карточка цвета и категории:

* он может сразу устно дать подсказку;
* остальные игроки могут сразу голосовать.

Никакая текстовая подсказка в систему не вводится.

## 8.2 Подтверждение ставок

Игрок:

* выбирает клетки;
* подтверждает свой набор ставок;
* после этого все еще может менять ставки, пока идет окно ожидания завершения.

## 8.3 Таймер 10 секунд

Таймер 10 секунд запускается, когда:

* **все неактивные игроки, у которых есть фишки и которые должны ходить, подтвердили ставки**

Во время этих 10 секунд:

* игроки **могут менять** свои ставки;
* после каждого изменения система должна оставлять таймер работающим, а не сбрасывать его, если не указано иное;
* по окончании 10 секунд голосование закрывается.

Если к моменту завершения окна:

* кто-то так и не поставил обязательную минимум одну фишку, его ход считается пропущенным.

### Важное проектное решение

Codex должен реализовать это как:

* фаза `voting_open`
* флаг `allPlayersConfirmedAt`
* отсчет 10 секунд на сервере
* клиенты получают realtime обновление оставшегося времени

## 12. Reconnect и идентичность сессии

Регистрация не нужна. Вход — по имени в комнате.

Нужна надежная схема возврата игрока после обрыва связи.

### Требование

Использовать:

* `playerId` / `hostId`
* `sessionToken`
* хранение токена в localStorage / secure cookie
* при повторном входе по коду комнаты клиент пытается восстановить сессию автоматически

Правила:

* имя в комнате уникально;
* если имя занято, но у клиента есть валидный sessionToken этого игрока — это reconnect;
* если имя занято и токен не совпадает — вход отклонять;
* хост должен переподключаться аналогично.

## 13. Подключение посреди игры

Если новый игрок вошел в уже идущую игру:

* получает стартовый банк фишек;
* добавляется в конец очереди активных игроков;
* может сразу участвовать как неактивный игрок, если это допустимо текущей фазой;
* не должен ломать уже идущий раунд.

Рекомендуемое правило:

* если игрок подключился во время активного раунда, он начинает участвовать **со следующего раунда**;
* в текущем раунде отображается как “joined_waiting_next_round”.

Codex должен реализовать это именно так, чтобы не ломать синхронизацию текущего голосования.

## 17. Правила reveal и анимаций

Важно разделить три этапа визуализации:

1. **Закрытое голосование**

   * никто не видит выборы других

2. **Reveal votes**

   * хост-экран показывает все ставки игроков
   * игроки могут видеть только собственный итоговый интерфейс или общий статус, но не обязаны видеть полную карту

3. **Reveal zone**

   * показывается зона 5 × 5
   * визуально классифицируются ставки:

     * miss
     * edge
     * near
     * center

4. **Apply results**

   * пересчет банков
   * обновление таблицы фишек

## 18. Генерация палитры

Codex должен реализовать отдельный сервис палитры.

Требования:

* палитра генерируется из seed;
* одна и та же seed дает одну и ту же палитру;
* палитра выглядит гармонично;
* клетки соседствуют плавно;
* палитра может немного меняться от партии к партии.

Рекомендуемый подход:

* использовать HSL/OKLCH-пространство;
* центр делать ближе к светлым/нейтральным тонам;
* по направлению к краям увеличивать насыщенность/цветовой разброс;
* финальные значения сохранять как `hex`.

## 21. Критически важные правила, которые Codex не должен нарушать

1. Хост видит поле всегда, игроки — нет.
2. Подсказка **не вводится в систему**, она устная.
3. Цвет видит только активный игрок.
4. Категорию видят все.
5. До завершения голосования никто не видит, кто куда поставил.
6. Одна клетка = максимум одна фишка от одного игрока.
7. Игрок обязан поставить минимум 1 фишку, если у него есть фишки.
8. Игрок может поставить не все фишки, а только часть.
9. Таймер 10 секунд стартует, когда все подтвердили, и в это время ставки еще можно менять.
10. Игрок с 0 фишек выбывает.
11. Новый игрок может войти в уже идущую игру, получает стартовый банк и добавляется в очередь.
12. Игра идет автоматически, хост не ведет раунды вручную.
13. Система должна быть полноценной, а не “MVP на один вечер”.

- язык интерфейса;
- seed палитры (опционально).

Локализация:
